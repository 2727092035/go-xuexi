# Go 60 天学习计划

目标：把原 90 天路线压缩到 60 天，但不减少应覆盖的知识范围。压缩方式不是砍内容，而是把每天的学习密度提高：每一天同时包含相关知识点、配套练习、项目应用和复盘验收。适合已有前端经验、希望用更高强度节奏进入 Go 后端开发实战的学习方式。

建议每天投入：

- 工作日：90 到 120 分钟
- 周末或空闲日：150 到 180 分钟
- 每天至少写 60 到 120 行 Go 代码

## 总体节奏

| 阶段 | 天数 | 内容 | 目标 |
| ---- | ---- | ---- | ---- |
| 第一阶段 | Day 1-10 | Go 语法、测试、并发基础 | 能独立写小型 CLI，覆盖函数、切片、map、struct、error、包、测试、goroutine/channel |
| 第二阶段 | Day 11-25 | Web API、数据库、认证 | 能写 HTTP API，连接 MySQL/PostgreSQL，完成 CRUD、认证、权限、文件、接口测试 |
| 第三阶段 | Day 26-40 | 工程化、中间件、部署 | 掌握项目分层、配置、日志、缓存、队列、性能、Docker、CI 基础 |
| 第四阶段 | Day 41-60 | 综合项目、进阶专题、复盘 | 完成可展示项目，并覆盖安全、性能、部署、文档、微服务入门和后续路线 |

## 每天固定动作

1. 10 分钟复盘昨天代码，先修一个小问题。
2. 40 到 60 分钟学习当天主知识点和关联知识点。
3. 60 到 90 分钟写基础练习、加餐练习或项目代码。
4. 20 到 30 分钟补测试、整理笔记、更新 `PROGRESS.md`。

每天不再只学一个孤立概念，而是尽量把新知识放进连续项目里使用。

## 每天内容交付标准

后续每天都要加强教学解释，尤其是 `demo/main.go` 和 `notes.md`：

- demo 必须按语法点分段，每段都有中文注释说明语法目的、核心写法和输出含义。
- demo 里只要出现新语法、新标准库函数、新方法调用或新写法，都要就近解释，不能默认初学者知道。例如 `strings.TrimSpace(raw)` 必须说明 `strings` 是标准库包、`TrimSpace` 是去掉前后空白、`raw` 是原始输入、返回值是清理后的字符串。
- notes 必须包含当天语法详解、demo 逐段解析、常见坑、JS/TS 对比、练习验收。
- notes 对每个新知识点必须写清楚：用途、参数、返回值、为什么这里要用、如果不用会出现什么问题。
- 练习必须从基础题到组合题递进：先练最小语法，再练接近业务的组合使用。
- 项目日也不能跳过语法说明，需要先解释当天用到的标准库、框架 API 或工程概念。
- 如果某个知识点第一次出现，要写“为什么需要它”和“什么时候不用它”。

## 压缩方式

60 天计划的原则是“内容不缩水，节奏更密集”。具体压缩方式：

- 把相关知识合并学习：例如 HTTP、JSON、REST、错误响应放在连续 2 天内完成。
- 每天同时安排基础练习和项目落地，避免只学概念。
- 每 5 到 7 天做一次重构和测试补齐，减少单独复盘日。
- 框架和工具只选主线深入，但会理解同类方案的差异，例如 Gin/Chi、GORM/sqlc、JWT/session。
- 微服务、Kubernetes、可观测性等进阶内容用最小可运行练习和方案设计覆盖，完整生产级落地放到后续深化。
- 并发、缓存、队列、安全、性能都必须在项目中有实际使用点。
